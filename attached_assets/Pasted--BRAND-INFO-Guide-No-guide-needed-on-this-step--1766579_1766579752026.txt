========================
LOCATIONS — Guide
========================

Before you connect
- Make sure your business details are accurate and up to date (name, category, address, phone, website, opening hours).
- If you have Unverified / Suspended / Duplicate locations in Google Business Profile (GBP), fix those first—otherwise some actions won’t publish.
- Assign a unique, standardized **store code** to every location (you’ll use the same store code in Sales & Inventory too).
- You’ll need someone who can grant **Owner / Primary Owner** access in GBP (Manager access cannot transfer ownership or manage user access).

Step 1 — Your Address Data
- Check consistency: Name–Address–Phone, category/attributes, opening hours (including special days), map pin location, website/social links.
- Confirm each location has a **unique store code** with a consistent format.

Step 2 — Where to Sync
- Select the platforms you want VenueX to sync your locations to (the options you see on screen).
- If you choose Apple, you must have an **Apple Business Connect** account (or be ready to create one).

Step 3 — Review & Connect
- After you connect GBP, VenueX will import your locations automatically.
- Reviews can be enabled right after import (in a single action).

Common gotchas
- New GBP users can be restricted for the first 7 days (some ownership/user actions may fail).
- If locations are unverified, you may be able to view reviews but not reply until verification is completed.

========================
OFFLINE SALES — Guide
========================

Before you begin
- Data delivery: VenueX ingests sales files via **SFTP/FTP**. During setup you’ll provide a file/folder URL.
- File format must be **CSV (UTF-8)**.
- Recommended frequency is **daily** (minimum weekly). Fresher data improves attribution quality.
- Personal identifiers (email/phone/name) must be **SHA-256 hashed on your side**. Do not send raw PII.

Step 1 — Your Offline Data
Required fields (minimum viable)
- Country (ISO code)
- Conversion Name
- Conversion Time (ISO-8601, include timezone)
- Conversion Value (numeric)
- Currency (ISO code)

Strongly recommended (to improve match rate)
- Email (lowercase + trimmed, then SHA-256)
- Phone (E.164 format, then SHA-256)

Helpful (if available)
- Store Code (must match your Locations store codes)
- Order ID (unique transaction id)

Step 2 — Build Mapping
- Make sure identifier normalization happens BEFORE hashing:
  - Email: lowercase + remove spaces
  - Phone: E.164 (+country code)
- Validate timestamps are ISO-8601 and values are numeric (no currency symbols).

Step 3 — Review & Save
- Start with a small test file to validate mapping and matching.
- You can backfill up to **90 days** of sales data (useful for pilots and validation).

========================
LOCAL INVENTORY — Guide
========================

Before you begin
- Local Inventory requires store-level availability. You must be able to export inventory by **Product ID + Store Code**.
- Product IDs must match the identifier used in your product catalog (e.g., Merchant Center alignment).
- Data delivery is the same standard as Offline Sales: **SFTP/FTP** + **CSV (UTF-8)**.
- Store Codes must match your Locations/Sales store codes exactly (same values + same format).

Step 1 — Your Inventory Data
Minimum fields
- Product ID (must match catalog identifier)
- Store Code
- Availability (in_stock / out_of_stock)

Optional (recommended if available)
- Price (if store price differs)
- Quantity

Step 2 — Build Mapping
- Confirm Product ID and Store Code formats match your catalog and location list exactly.
- Ensure availability values are standardized (in_stock / out_of_stock).

Step 3 — Review & Save
- Daily updates are strongly recommended to keep availability accurate.
