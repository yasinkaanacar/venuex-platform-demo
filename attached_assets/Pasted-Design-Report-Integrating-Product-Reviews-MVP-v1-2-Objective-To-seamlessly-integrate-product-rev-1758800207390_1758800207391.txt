Design Report: Integrating Product Reviews (MVP v1.2)
Objective: To seamlessly integrate product reviews (from website/app) into the existing MVP workflow without significant architectural changes, preserving the platform's simplicity and speed.

1. Core Principle: A Single, Context-Aware Workspace
Instead of creating a separate product review inbox, we will enhance the existing Gelen Kutusu (Inbox) to intelligently switch between two contexts: Locations and Products. This maintains the "single pane of glass" philosophy and requires the fewest UI changes.

2. The Migration Plan: Minimalist Enhancements
We will make three surgical changes to the existing design.

Action 1: Introduce a "Review Source" Filter
Positioning: This new filter will be placed at the very top of the filter bar on the Inbox page, to the left of the Location Filter. It becomes the primary choice the user makes.

Component Type: A pill-style button group, consistent with the existing Reply Status filter. This provides a clear, high-visibility toggle.

Functionality & Styling:

Two buttons: Locations and Products.

Default State: Locations will be the default selection when the page loads.

Active State: The selected button will have a high-contrast style (dark background, white text) to clearly indicate the current context of the entire inbox.

Action 2: Make the Filter Bar "Context-Aware"
Functionality: The rest of the filter bar will dynamically change based on the "Review Source" selection.

When Locations is selected: The filter bar will function exactly as it does now, displaying the Location Filter dropdown.

When Products is selected:

The Location Filter dropdown will be hidden or disabled.

A new Product Filter dropdown will appear in its place.

Component Type: A standard dropdown menu.

Options: "All Products," followed by a list of product names (e.g., "Running Shoes," "Winter Coat," "Coffee Maker").

This is the key to the minimal-change approach. We are swapping one component for another within the same layout, not redesigning the page.

Action 3: Make the Review Card "Context-Aware"
Functionality: The content displayed on each review card in the inbox list will adapt to the selected source.

When Locations is selected: The card design remains unchanged, showing the Location Name with a map pin icon.

When Products is selected:

The Location Name and map pin icon are replaced by the Product Name and a small product icon (e.g., a shopping tag fa-tag).

An SKU or Product ID can be displayed in smaller text below the product name for clarity.

All other elements of the card (Customer Name, Rating, Review Text, Analytics Tags) remain identical. The reply functionality is also completely unchanged.

3. What Doesn't Change (Preserving Simplicity)
✅ The Özet (Overview) Tab: This page will remain focused on location-based analytics for the MVP to avoid scope creep. The "Avantajlar/Dezavantajlar" module will not be affected.

✅ The Reply Panel: The slide-in panel for writing a response is universal. Its functionality does not need to change whether the user is replying to a location review or a product review.

✅ The Core Layout: We are not adding new pages or significantly altering the structure. We are only adding one new control and making existing components smarter.

Example User Workflow (E-commerce Manager)
An E-commerce Manager wants to find and respond to negative feedback about the new "Winter Coat" that was just launched on the website.

Select Source: The manager navigates to the Inbox tab and clicks the new Products button in the "Review Source" filter. The page instantly switches to product review mode.

Filter by Product: The Location Filter is now replaced by a Product Filter. The manager selects "Winter Coat" from the dropdown.

Filter by Rating: They select "1-2 Stars ★" from the Filter by Rating dropdown to see the most critical feedback.

Act on Insight: The inbox now shows only the 1 and 2-star reviews for the Winter Coat. The manager sees a review tagged with "Sizing Issue," clicks on it, and writes a helpful reply directly from the same interface they are already familiar with.

This approach successfully adds significant new functionality while fully respecting the constraint of not changing the page or its core mechanics "too much."
